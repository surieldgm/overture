# Pruebas virtuales con personas — 2026-05-07

Pruebas exploratorias del wizard web de Overture (`python -m overture ui`) ejecutadas
con tres personas naive, una por sesión. El objetivo es identificar fricciones
de UX que informen los próximos pasos de producto.

## Metodología

- **Interfaz probada:** wizard web local, una persona por puerto
  (`localhost:8765` Carla, `8766` Tomás, `8767` Rocío).
- **Tooling:** Claude Preview MCP — screenshots y accessibility snapshots de cada
  pantalla, sin acceso al código fuente desde la perspectiva del operador.
- **Estado de partida:** servidor recién arrancado por persona, store-dir vacío
  (`/tmp/overture-test-<persona>`).
- **Codex CLI:** ausente del PATH del proceso. Carla recibió un servidor
  reiniciado con `OVERTURE_LLM_CLIENT=fake` después de detectar la fricción
  (recuperación plausible para su perfil técnico). Tomás y Rocío corrieron
  contra el servidor sin ese fallback — fiel a su perfil.
- **Restricción del operador:** sólo se actuó con la información que cada persona
  podría tener (sin lecturas de README, walkthrough, o código). Cuando una
  persona se atascó genuinamente, el test terminó ahí.
- **Idea de producto** (inventada por persona, coherente con su perfil):
  - Carla: *Add session metadata to the peer onboarding template…*
  - Tomás: *Show a sidebar of recent intakes on the wizard…*
  - Rocío: *Send a weekly digest email to onboarded users…*

## Test 1 — Carla Mendoza (Designer #1)

### Setup

- Email: `carla@dogfood.test`
- Servidor: `python3 -m overture ui --port 8765 --store-dir /tmp/overture-test-carla`
  (después del primer fallo, reiniciado con `OVERTURE_LLM_CLIENT=fake`)
- Idea: *"Add session metadata to the peer onboarding template so Designer #1
  can leave timestamps, tools used, and example screen recordings for Designer
  #2 instead of handing off a generic empty form."*

### Pasos observados

1. Navega a `/intake`. El servidor redirige a `/auth/login?next=/intake`.
2. Ve la pantalla **"Designer sign in"** y el botón "Send magic link". Mete
   email, submitea.
3. Llega a `/auth/magic-link` ("Magic link sent"). Encuentra el link
   `Open development magic link` y lo clickea (después de un primer click
   fallido sobre breadcrumbs).
4. Aterriza en `/intake`. Lee el textarea "Raw idea", el contador 0/5,000 y
   ve el botón "Start research approval". Ignora el panel lateral "Examples".
   Escribe su idea, submitea.
5. **Primera vez**: la página `/research/approval` muestra:
   > Codex CLI executable not found on PATH. Install the Codex CLI, set
   > OVERTURE_CODEX_EXECUTABLE, or run with OVERTURE_LLM_CLIENT=fake for
   > deterministic local research.
6. Carla (técnica) reconoce las opciones, mata el server y lo relanza con
   `env OVERTURE_LLM_CLIENT=fake python3 -m overture ui …`. Reingresa, redo
   intake.
7. **Segunda vez**: research approval muestra dos fuentes sugeridas
   (`example.test/symphony-ticket-evidence`, `example.test/designer-intake-research`),
   ambas pre-marcadas como "Approve". Carla las acepta y submitea.
8. `/research/complete` confirma "Saved 2 approved sources for idea_…" pero
   no ofrece acción siguiente. Carla hace click en la nav superior `Synthesis`.
9. `/synthesis` carga el cached brief. Ve secciones Problem, User need,
   Evidence, Claims (vacío), Assumptions, Connected concepts, Proposed
   capability, Candidate ticket. Click en "Continue to ticket review".
10. `/ticket` muestra **"Placeholder for Symphony ticket drafting"** + un
    textarea vacío + el error inline "ticket is missing a level-one title".
    No se generó draft.
11. Carla se desvía a `/peer-onboarding` (su rol como Designer #1 lo demanda).
    Ve el artifact ya escrito por `designer_1` en modo lectura. No hay botón
    para crear o editar. Termina sesión.

### Monólogo interno (extracto)

> "El wizard reconoce que falta Codex y me da la salida (`LLM_CLIENT=fake`).
> Bien. Pero tuve que reiniciar el server desde la terminal — alguien más
> técnico que yo no podría hacer eso. […] El brief de synthesis es muy
> templated; la sección 'User need' dice algo genérico que no es mi idea.
> El título del candidate ticket está cortado a la mitad. […] El paso `/ticket`
> es un placeholder. Eso es lo que esta tool promete generar. […] Como
> Designer #1, esperaba poder *escribir* mi peer onboarding template, pero
> el wizard sólo me deja leer el de otro designer."

### Fricciones encontradas

| # | Severidad | Descripción |
|---|---|---|
| 1 | Media | Login forzoso al entrar a `/intake` sin aviso ni opción "soy local, sáltame esto" |
| 2 | Baja | Copy "Development outbox" + "magic-links.jsonl" usa jargon de developer |
| 3 | Alta | Server sin Codex no se recupera desde la UI; fix exige reiniciar con env var |
| 4 | Media | Las dos fuentes sugeridas vienen pre-marcadas como Approve, en tensión con el principio "aprueba sólo lo que inspeccionaste" |
| 5 | Media | URLs `example.test/...` no son verificables, pero el flujo de aprobación las trata como reales |
| 6 | Baja | `/research/complete` no tiene botón "Continue to synthesis" — es callejón |
| 7 | Media | Synthesis brief en modo fake es mayormente templated; no refleja la idea específica |
| 8 | Baja | Candidate ticket title cortado: "…Designer #1 can" (oración incompleta) |
| 9 | **Crítica** | `/ticket` es placeholder; no genera el draft desde el brief. Rompe la promesa core del producto. |
| 10 | Alta | `/peer-onboarding` es viewer read-only de un artifact pre-existente; no hay flujo de creación/edición |

### Resultado

Carla completó hasta synthesis pero **no logró generar un ticket** desde la UI.
Encontró 2 hallazgos críticos/altos por revisar en backlog.

## Test 2 — Tomás Iriarte (Designer #2)

### Setup

- Email: `tomas@overture.test`
- Servidor: `python3 -m overture ui --port 8766 --store-dir /tmp/overture-test-tomas`
  (sin Codex, sin fallback de env var)
- Idea: *"Show a sidebar of recent intakes on the wizard so I can reuse
  phrasing from past ideas instead of starting from a blank page every time."*

### Pasos observados

1. Navega a `/intake`. Login → Magic link sent. Ignora la línea "Development
   outbox: …" porque le suena a otra cosa, pero ve el link verde y lo clickea.
2. Aterriza en `/intake`, escribe su idea, submitea.
3. `/research/approval` muestra el mismo error de Codex que vio Carla.
4. Tomás lee las tres opciones de remediación:
   - "Install the Codex CLI" — sin link, no sabe qué Codex.
   - "set OVERTURE_CODEX_EXECUTABLE" — no sabe qué es un environment variable.
   - "run with OVERTURE_LLM_CLIENT=fake" — entiende vagamente que es un
     parámetro, pero el server ya está corriendo y él no lo lanzó.
5. Por desesperación, clickea `Synthesis` en la nav superior para ver si
   puede saltar el paso.
6. `/synthesis` muestra: "Placeholder for synthesis brief review.
   **Research result not found for idea_6f09ab10d13e58519e5a92fdf9ef817f.**"
7. **Abandona.** Mensaje mental: "Le pregunto a Carla por Slack."

### Monólogo interno (extracto)

> "Tres opciones para resolver, y ninguna la puedo hacer yo desde aquí. Y
> 'Placeholder for synthesis brief review' me confirma que esta cosa está
> incompleta. Mejor escribo en Slack."

### Fricciones encontradas

| # | Severidad | Descripción |
|---|---|---|
| 11 | **Crítica** | Error de Codex es intécnicamente correcto pero **inactivable desde la UI** para semi-técnicos |
| 12 | Alta | El wizard no degrada con elegancia: saltar de paso devuelve "Research result not found" en vez de "vuelve atrás y haz X" |
| 13 | Baja | Dos páginas (Synthesis y Ticket) tienen literal "Placeholder for…" — comunican producto inacabado |

### Resultado

Tomás **abandonó en Research**. No produjo ningún artefacto. Findings 11 y
12 son los que más le pegan a su perfil.

5. Escribe su idea (largueña, con contexto de PM), submitea.

7. Antes de rendirse, prueba clickear `Ticket` en la nav (es lo que ella

> "¿Por qué dice 'designer' si me lo recomendaron como tool de producto?

### Fricciones encontradas

| # | Severidad | Descripción |
|---|---|---|
| 14 | Media | Heading "Designer sign in" + copy "designer idea" alienan a founders/PMs no-diseñadores |
| 15 | Alta | Magic link UX asume que el usuario entenderá "Development outbox" — incomprensible para no-técnicos |

## Síntesis cross-test

### Lo que funciona

- **Login con magic link funciona** una vez que la persona descubre el
  fallback de dev. Tres de tres lo lograron.
- **Intake es claro**: el textarea, el contador, y el botón comunican qué
  hacer. Tres de tres entendieron sin instrucciones.
- **Synthesis (cuando hay brief)** comunica estructura: secciones nombradas,
  brief read-only con disclosure, botón explícito para continuar.
- **Peer-onboarding como viewer** está bien diseñado para Designer #2 que
  recibe el handoff. La tipografía y la jerarquía son legibles.

### Patrones de fricción que cruzan personas

1. **Lenguaje técnico en lugares de UX general.** "Development outbox",
   "executable not found on PATH", "Ticket Markdown", "Research result not
   found for idea_<hash>". Estos textos asumen que el lector es developer.
2. **El producto promete idea→ticket pero rompe en `/ticket`.** Es el paso
   que justifica todo el flujo, y es un placeholder. Dos de tres personas
   llegaron y vieron exactamente lo mismo.
3. **No hay degradación graceful entre pasos.** Saltar de un paso roto al
   siguiente devuelve un error técnico, no una guía.
4. **Identidad sólo "designer".** El wizard se llama "Designer sign in",
   pide "the designer idea", y el peer transfer es "Designer #1/#2". El
   producto se vende como herramienta de ideas → tickets, pero el copy se
   resiste a reconocer otros perfiles.
5. **Pre-aprobación de sources viola el principio del producto.**
   `docs/symphony-ready-ticket-schema.md` enfatiza "approve only what you
   inspected". La UI las pre-marca todas como Approve.

### Bloqueadores absolutos

- **Codex CLI dependency sin fallback en UI**: dos de tres personas se
  atoraron permanentemente aquí. Es el primer punto de pérdida masiva.
- **`/ticket` placeholder**: incluso para usuarios técnicos, romper la
  promesa del producto en su paso final hace que el wizard no sea utilizable
  end-to-end desde la UI.

## Recomendaciones priorizadas para los siguientes pasos

### P0 — Desbloquea casos de uso completos

1. **Wirear `/ticket` al synthesis cache**. La página debe pre-llenar el
   textarea con el draft generado desde el brief (lo que hace `overture run`
   en CLI). Hoy el usuario lo ve vacío.
2. **Fallback de Codex en la UI**. Cuando el server detecta que Codex no
   está, ofrecer un toggle "Use deterministic local mode (no LLM)" en la
   misma página de research approval — sin requerir reiniciar el server. O,
   menos invasivo: detectar al arrancar y elegir fake automáticamente,
   imprimiendo un banner "Modo determinista activo".

### P1 — Reduce fricción de cuatro/cinco fricciones medias

3. **Remover pre-aprobación** de sources sugeridas. Default = sin elegir;
   force decisión consciente.
4. **Botón "Continue to <next>"** en cada confirmación (`/research/complete`,
   `/auth/magic-link` después del consume, etc.).
5. **Genericizar copy "designer"**. Sustituir "Designer sign in" por
   "Sign in", "designer idea" por "your idea". La estructura interna
   (Designer #1/#2 en peer onboarding) puede quedarse porque ahí es
   correcta semánticamente.
6. **De-jargon del magic-link**. Esconder "Development outbox" detrás de
   un `<details>` con label "Local development details" o similar.

### P2 — Pulido y completitud

7. **Fix truncation** del candidate ticket title (Test 1, finding #8).
8. **Crear flow de edición** del peer onboarding template para Designer #1
   (Test 1, finding #10). El esquema lo soporta; la UI no expone la
   capacidad.
9. **Mejorar empty-state de synthesis**: cuando no hay brief, redirigir o
   mostrar "Necesitas completar Research primero — [Volver a Research]".
10. **No mostrar el hash del intake** como id principal en URLs/UI; usar
    un nombre derivado de la primera frase.

### Investigación adicional sugerida

- Repetir los 3 tests **con Codex realmente instalado**. Las observaciones
  sobre el brief templated y el title truncation pueden ser efectos
  secundarios del modo fake. Necesario validar que el flujo "feliz" con
  Codex real produce contenido específico de la idea.
- **Test con persona "Symphony"** (el agente downstream): tomar el draft
  generado por Carla y dárselo a un LLM-coding-agent. ¿Puede arrancar la
  implementación sin preguntar? Eso valida la pieza más downstream del
  contrato.
- **Test del flujo CLI** con las mismas tres personas. Para Carla podría ser
  más rápido; para Rocío es probable que sea inutilizable. Confirmar.

## Apéndice — comandos de reproducción

Cada test es reproducible:

```sh
mkdir -p /tmp/overture-test-carla
python3 -m overture ui --port 8765 --store-dir /tmp/overture-test-carla
```

Cambia el puerto y store-dir para Tomás (`8766`) y Rocío (`8767`). Si Codex no
está en el PATH, el server arranca con un warning y usa research determinístico
automáticamente (ver siguiente sección).

## Addendum — fixes incluidos en este PR

Después de los tests, se cerraron las dos fricciones más bloqueantes que
afectaban a Tomás y Rocío:

- **Finding #3 / #11 / #16 — Codex CLI ausente como muro absoluto**: ahora
  resuelto. El comando `python -m overture ui` (y `python -m overture
  research`) detecta al arranque si Codex está disponible y, si no, hace
  fallback automático al cliente determinístico imprimiendo un warning a
  stderr. Sin necesidad de reiniciar con `OVERTURE_LLM_CLIENT=fake`.
  Cambios en [`overture/research_llm.py`](../../overture/research_llm.py)
  (helper `codex_cli_available`),
  [`overture/cli.py`](../../overture/cli.py)
  (`_select_research_llm_client`), y tests en
  [`tests/test_research_llm.py`](../../tests/test_research_llm.py).
- Validado con un re-test de Carla en Preview, server arrancado sin env vars,
  flujo intake → research → synthesis OK.

Los demás findings (placeholder en `/ticket`, peer-onboarding viewer-only,
copy "designer-céntrico", pre-aprobación de sources, etc.) **no** se
abordan en este PR — están listados en las recomendaciones priorizadas para
los siguientes pasos.
